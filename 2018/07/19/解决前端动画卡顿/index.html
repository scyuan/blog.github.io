解决前端动画卡顿

发表于 2018-07-19 | 分类于 CSS |

前言

在一个项目中的首页上，首页通过改变left的值以及transition来实现动画的过程。在实际体验中，动画切换体验有点差，因为能感觉到明显的卡顿效果。这个原因是怎样引起的？然后怎么解决这个问题呢？（PC和移动端都会有这个问题）

优化前代码

vue中点击事件的代码

总共有5个div，当点击某个div时，通过index获取下标，然后通过css方法对改变5个元素的left值。从而达到移动的效果。

上述代码不管是在chrome还是在移动端（iphone，锤子）会出现卡段的情况。百思不得其解，觉得现在硬件性能很强大了，不可能就单单5个div的动画效果都会出现卡顿呀。后来参考文章，将改变left修改为通过transform:translate()来改变div的位置。方法是可行的。

优化后

改成transform后（通过使用translate也开启了GPU硬件加速），发现已经没有卡顿效果了。

优化前后性能对比

针对同一个动画进行比较，打开chrome控制台的performance查看页面性能情况。每秒60帧是最适合人眼的交互帧率。低于60帧的情况人会明显感觉卡顿。

🔽优化前从控制台Frames可以看出当前动画的帧率，大概维持在30帧-70帧的情况，帧率波动大，并且出现了帧率低于60帧的情况。rendering和painting所花的事件分别为80ms和59ms。

🔽优化后帧率维持在50帧-70帧的水平，没有感觉到明显的卡顿，rendering和painting所花的事件分别为45ms和54ms。可以看出rendering优化了很多。

原因

现代的浏览器通常会有两个重要的执行线程，这2个线程协同工作来渲染一个网页：

主线程负责：

  1. 运行JavaScript。
  2. 计算HTML 元素的 CSS 样式。
  3. 页面的布局
  4. 将元素绘制到一个或多个位图中
  5. 将这些位图交给合成线程

合成线程负责：

  1. 通过 GPU将位图绘制到屏幕上
  2. 通知主线程更新页面中可见或即将变成可见的部分的位图
  3. 计算出页面中哪部分是可见的
  4. 计算出当你在滚动页面时哪部分是即将变成可见的
  5. 当你滚动页面时将相应位置的元素移动到可视区域

优化前（transition:left），整个过程如下：不断循环这个过程，直到transition动作完成。红色框可能比较慢，蓝色框动作很快。浏览器需要去做大量的工作，每一帧的内容都需要重新布局和重新绘制，然后去告诉合成线程。这样当然会很慢。

优化后（animation）,可以看出动画都交给了GPU来做（GPU是很快的），浏览器只需要告诉合成线程去执行哪个动画就可以了。

根据定义，CSS 的transform属性不会更改元素或它周围的元素的布局。transform属性会对元素的整体产生影响，它会对整个元素进行缩放、旋转、移动处理。

这对浏览器来说是个好消息 ！浏览器只需要一次生成这个元素的位图，并在动画开始的时候将它提交给GPU去处理 。之后，浏览器不需要再做任何布局、 绘制以及提交位图的操作。从而，浏览器可以充分利用 GPU 的特长去快速地将位图绘制在不同的位置、执行旋转或缩放处理。

总结

  1. 尽量使用animation来完成动画（过渡），避免使用left,width,height等
  2. 开启GPU硬件加速（在实际体验装好像效果不是很明显…），但避免过度使用。
